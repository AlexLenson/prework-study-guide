HTML

  • The head element contains information about the webpage.
  • Head element contains metadata that is not visible to the user
  • Metadata tells the browser how to render the page correctly
  • Title tag is visible to user (good title name is important for SEO!)
  • The body element represents the visible content shown to the user.
  • Body element is an example of a semantic HTML element. Semantic elements are element names that describe their content, use them often!
  • Common non-semantic elements include "div" (to create division on page) and "span" (inline division)
  • The "alt" attribute for the "img" element describes the image and is important for screen readers/SEO
  • The body element in "index.html" for this page is structured using "header", "main" and "footer" elements. The "main" element is divided into four sections using "section" elements.
  • Right now we are in the "section" element with attribute id="html-section"

Git

  • Git is used to track versions of a project
  • GitHub allows you to store your project in a remote repository and collaborate with other developers
  • GitHub Workflow:
    • Make new feature branch (locally)
      • This is where we can save our "work in progress". Good practice to name branches "feature/(whatever the feature is)"
    • Add & commit changes (locally)
    • Before pushing, it's good practice to pull source code from github to ensure we have latest commits and changes & no conflicts bw GitHub and local repo:
      • "git pull origin main" (should say "Already up to date")
    • Push changes to feature branch on github (remote):
      • "git push origin feature/starter-code"
      • Note: Always push to your feature branch, not main branch. Essential for keeping bugs from breaking project
    • Can now see the repo we pushed changes to in github
    • Open PR (github)
      • Make sure correct base and compare branches are selected ("base: main" and "comare: feature/starter-code")
    • Merge into main branch
    • Main branch is now updated with your changes
    • Notes:
      • Don't code directly in main branch. Instead, make a copy (feature branch) and merge back into main branch.
      • Changes made to main branch tracked by feature branches. Main branch (should be) the final, most updated version of the application.
  • git status: checks what branch we are currently on
  • git checkout -b branch-name: creates a new branch and switches to it
